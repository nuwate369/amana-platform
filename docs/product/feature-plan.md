# خطّة المزايا — أمانة

وثيقة تحليل أعمال وتخطيط تنفيذي. تُحدَّث مع كل سبرنت.

---

## قاعدة الأطراف الثلاثة (ملزمة)

> السبب الجذري لأعطالنا السابقة أنّ المهامّ كانت تُعرَّف من جانب واحد
> («أضف زرّ إنهاء الرحلة») فتُبنى في تطبيق واحد، ويكتشف المستخدم أنّ الطرف
> الآخر لا يعلم بها. الرحلة عقد بين طرفين، لا حدثًا في تطبيق.

**لا تُعتبر أيّ مهمّة مكتملة قبل الإجابة على الخمسة:**

| # | السؤال |
|---|---|
| ١ | ماذا ترى **الراكبة**؟ |
| ٢ | ماذا ترى **السائقة**؟ |
| ٣ | ماذا ترى **الإدارة**؟ |
| ٤ | ما **عقد البيانات** الذي يربطهم؟ (جدول/عمود/إشعار) |
| ٥ | ماذا يحدث إذا **فشل** أحد الأطراف؟ (أغلقت التطبيق · نفد الشحن · رفضت) |

سؤال بلا إجابة ⇒ المهمّة مرفوضة، لا تدخل السبرنت.

## الوضع الحالي

**مكتمل:** التوزيع · التحديث الفوري (OTA) · الإشعارات الفورية · دورة الرحلة الأساسية
(طلب ← قبول ← وصول ← بدء ← إنهاء) · شريط الرحلة الجارية · الدفع المعلّق ·
لوحة الإدارة · الأمان.

**الأساس غير المكتمل** (بُنيت قاعدة بياناته ولم تُبنَ واجهاته):
تأكيد الركوب · الدفع النقدي · المحفظة · تفاصيل الرحلة · مهلة تحرير السائقة.

**المزايا المميِّزة:** لم يُبنَ منها شيء. وهي التي تقوم عليها دراسة الجدوى.

# السبرنت ١ — إغلاق الأساس

**الهدف:** رحلة كاملة بلا فجوة، من الطلب إلى الدفع.
**لماذا أوّلًا:** ميزة مميِّزة فوق أساس مكسور لا قيمة لها. ولجنة التحكيم
ستجرّب الرحلة قبل أن تنظر إلى الذكاء الاصطناعي.

### م-١ · تأكيد الركوب

| الطرف | ما يراه |
|---|---|
| 🟣 الراكبة | عند `arrived`: زرّان — **«ركبتُ»** / **«لم أجد سائقتي»** |
| 🔵 السائقة | «بانتظار تأكيد الراكبة» — زرّ «بدء الرحلة» **معطّل** حتى يصل |
| ⚫ الإدارة | عمود «تأكيد الركوب» في الرحلات الحيّة |

**عقد البيانات:** `rides.boarding_confirmed_at` (موجود) — تكتبه الراكبة، تقرؤه السائقة لحظيًّا.

**السيناريوهات:**
1. تؤكّد ← `in_progress` ← إشعار للسائقة
2. «لم أجد سائقتي» ← `no_show` ← بلا رسوم ← إشعار للطرفين
3. **لا تؤكّد ٥ دقائق** ← تظهر للسائقة «بدء الرحلة يدويًّا» مع تسجيل السبب
4. تغلق التطبيق ← الشريط في الرئيسية يعرض «سائقتك بانتظارك — أكّدي ركوبك»

**معيار القبول:** لا يمكن أن تبدأ رحلة والراكبة لم تركب — إلا بتجاوز يدوي مسجَّل.

### م-٢ · الدفع النقدي والمحفظة

| الطرف | ما يراه |
|---|---|
| 🟣 الراكبة | تختار **نقدًا/بطاقة عند الطلب** · شاشة الدفع تعرض الرصيد المطبَّق · تؤكّد أو تعترض |
| 🔵 السائقة | بعد الإنهاء: «كم استلمتِ نقدًا؟» ← تُدخل المبلغ ← تُقفل |
| ⚫ الإدارة | تقرير التسويات · فروقات `settlement_diff` · أرصدة المحافظ |

**عقد البيانات:** `settle_ride()` (جاهزة) · `wallet_transactions` (جاهز) · `rides.payment_method`

**السيناريوهات:** (مصفوفة متّفق عليها)
1. الأجرة ٤٥ · دفعت ٤٥ ← مطابق
2. الأجرة ٤٥ · دفعت ٥٠ ← **+٥ رصيد للراكبة**
3. الأجرة ٥٥ · دفعت ٥٠ ← **تُقفل عند ٥٠**، الفارق للإحصاء فقط، **لا دَين**
4. رصيد ٥ + أجرة ٥٥ ← المطلوب نقدًا ٥٠
5. الراكبة تعترض ← تذكرة دعم · المبلغ معلّق
6. لا تؤكّد خلال ٦٠ ثانية ← تُقفل تلقائيًّا (لئلّا تعلق السائقة)

**معيار القبول:** الرصيد لا يهبط تحت الصفر أبدًا. كل ريال متتبَّع إلى رحلته.

### م-٣ · مهلة تحرير السائقة

**المشكلة:** السائقة تنتظر دفعًا قد لا يأتي، فلا تستقبل طلبات جديدة.
**الحل:** بعد ٥ دقائق من `completed` تُحرَّر تلقائيًّا. الدَّين لا يُسجَّل (قرار متّفق عليه)، والرحلة تبقى «بانتظار الدفع» عند الراكبة.

### م-٤ · شاشة تفاصيل الرحلة

الضغط على أيّ رحلة في السجلّ يفتح: كل الأوقات (قُبلت · وصلت · بدأت · انتهت ·
دُفعت) · طريقة الدفع · الرصيد المطبَّق · المسار · التقييمات · زرّ دعم.
**نفس الشاشة للطرفين** بمنظورين.

### م-٥ · مراجعة الترجمة

نقل كل نصّ عربي مثبَّت إلى `@amana/i18n`. ست شاشات مصابة.

# السبرنت ٢ — أوّل ميزتين مميِّزتين

**لماذا هاتان أوّلًا:** أعلى ربط بين الطرفين وأقلّ جهد — تُثبتان أنّ قاعدة
الأطراف الثلاثة تعمل قبل الميزات الأثقل.

### م-٦ · 🛡️ التقييم المزدوج

**القيمة:** حماية متبادلة — والسائقة ترى تقييم الراكبة **قبل** القبول.

| الطرف | ما يراه |
|---|---|
| 🟣 الراكبة | تقييمها في حسابها · تعرف أنّه ظاهر للسائقات |
| 🔵 السائقة | **تقييم الراكبة داخل بطاقة الطلب قبل القبول** · «موثوقة» |
| ⚫ الإدارة | متوسّطات الطرفين · تنبيه عند هبوط تقييم راكبة |

**عقد البيانات:** `ratings` (موجود) + دالّة `passenger_rating(uuid)` + عرضها في بطاقة الطلب.

**السيناريوهات:**
1. راكبة بلا تقييمات ← «جديدة» لا «سيّئة»
2. تقييم < ٣ ← تحذير للسائقة لا منع
3. تقييم < ٢ + ٥ رحلات ← مراجعة إدارية
4. السائقة ترفض ← يُسجَّل السبب (بلا كشف هويّتها للراكبة)

### م-٧ · 👩‍👧 مرافقة الأطفال

**القيمة:** خدمة لا يقدّمها أيّ منافس. رسوم إضافية ١٠ ريالات.

| الطرف | ما يراه |
|---|---|
| 🟣 الراكبة | خيار «مع طفل» عند الطلب · عدد المقاعد · +١٠ ريالات ظاهرة |
| 🔵 السائقة | **شارة «مقعد أطفال» في ملفّها** (تُوثَّق من الإدارة) · الطلب يصلها فقط إن كانت مؤهَّلة |
| ⚫ الإدارة | توثيق تأهيل السائقة · تقرير الطلبات |

**عقد البيانات:** `drivers.child_seat_certified` · `rides.child_seats` · التوزيع يُصفّي السائقات.

**السيناريوهات:**
1. لا سائقة مؤهَّلة قريبة ← «قد يطول الانتظار — أتودّين بلا مقعد؟»
2. سائقة مؤهَّلة رفضت ← يُعاد للتوزيع
3. توثيق التأهيل مرفوض ← الشارة تُزال والطلبات تتوقّف
4. الطفل بلا مقعد فعليًّا ← السائقة تُبلغ ← تذكرة

# السبرنت ٣ — الاقتصاد والذكاء

### م-٨ · 📈 التسعير الديناميكي
مؤشّر طلب (منخفض/متوسّط/مرتفع) يُحسب من نسبة الطلبات إلى السائقات المتاحة
في المنطقة. **الراكبة** ترى المؤشّر والسعر قبل التأكيد · **السائقة** ترى
«ذروة — دخل أعلى» · **الإدارة** تضبط المعاملات وحدودها.
> ⚠️ سقف إلزامي (مثلًا ×١٫٨) وشفافية كاملة — وإلّا تحوّلت لشكوى.

### م-٩ · 🧠 مساعد السفر الذكي
تكتب الراكبة مزاجها → اقتراح وجهات بنسبة تطابق → «اختاري هذه الوجهة» تملأ
الوجهة في شاشة الطلب. البنية جاهزة (Groq عبر لوحة الإدارة).
**السيناريوهات:** تعذّر الخدمة ← اقتراحات ثابتة · نصّ غير مفهوم ← سؤال توضيحي
· الوجهة خارج نطاق التغطية ← تنبيه قبل الطلب.

### م-١٠ · 🌿 البصمة الكربونية
تُحسب من مسافات الرحلات التشاركية · نقاط خضراء · «زرعتِ ٣ أشجار».
**مطلب مصداقية:** معادلة الحساب معلنة، والأشجار شراكة حقيقية أو تُسمّى «رمزية».

# السبرنت ٤ — الأثقل

### م-١١ · 🔒 المجموعات المغلقة + الرحلات التشاركية
مجموعة خاصّة (زميلات عمل/دراسة) · رحلة تشاركية داخلها · تقسيم الأجرة ·
سائقة واحدة لعدّة راكبات.
> ⚠️ **الأعلى تعقيدًا:** عدّة راكبات في رحلة واحدة تكسر افتراض «راكبة واحدة
> لكل رحلة» في كل الجداول. يحتاج تصميم بيانات منفصلًا قبل البدء.

### م-١٢ · 🗺️ السياحة
وجهات سياحية · حجز جولة. يعتمد على م-٩.

## الترتيب والاعتماديات

```
السبرنت ١ (أساس)  ──►  السبرنت ٢ (تمايز)  ──►  السبرنت ٣  ──►  السبرنت ٤
     م-١ ← م-٢ ← م-٣            م-٦ · م-٧          م-٨ · م-٩ · م-١٠   م-١١ ← م-١٢
```

- م-٢ يعتمد على م-١ (لا دفع قبل رحلة صحيحة)
- م-٧ يعتمد على م-٦ (كلاهما يعدّل بطاقة الطلب)
- م-١٢ يعتمد على م-٩
- م-١١ مستقلّ لكنّه الأثقل — يُؤجَّل عمدًا

## تقدير الجهد

| السبرنت | المهامّ | التقدير |
|---|---|---|
| ١ | م-١ … م-٥ | ~٦ ساعات |
| ٢ | م-٦ · م-٧ | ~٥ ساعات |
| ٣ | م-٨ · م-٩ · م-١٠ | ~٨ ساعات |
| ٤ | م-١١ · م-١٢ | ~١٢ ساعة |

> كلّها JavaScript + SQL ⇒ تصل بـ `eas update` بلا بناء ولا إعادة تثبيت.

## معيار «منجَز» لكل مهمّة

- [ ] أُجيب على أسئلة الأطراف الثلاثة الخمسة
- [ ] بُنيت الأطراف المعنيّة **كلّها** في نفس المهمّة
- [ ] كل سيناريو استثناء مُعالَج (لا «سيحدث لاحقًا»)
- [ ] `typecheck` نظيف في التطبيقات الثلاثة
- [ ] نصوص مترجمة لا مثبّتة
- [ ] جُرّبت على جوالين حقيقيّين — لا محاكي

# وثيقة تصميم مشروع "أمانة" — مرحلة التصميم (Design Phase)

> هذه الوثيقة تُبنى على الوثيقة التأسيسية للمشروع، وتُركّز على **مرحلة التصميم** قبل الانتقال للبرمجة: توصيف كل تطبيق، برومبت احترافي لتوليد التصميم عبر Stitch، وشرح مفصل لكل شاشة. نظام التسجيل المعتمد في هذه المرحلة: **البريد الإلكتروني وكلمة المرور**.

---

## 1. نظرة عامة على الأنظمة الثلاثة

**هذه ثلاثة تطبيقات منفصلة تمامًا**، لكل منها ملف تشغيل (build) وواجهة مستقلة، لا تطبيق واحد بأدوار متعددة. السبب: تجربة المستخدم مختلفة جذريًا بين راكبة وسائقة (السائقة تحتاج ملاحة وإدارة أرباح، والراكبة تحتاج طلب وتتبع)، ولأسباب أمنية وتقنية (فصل صلاحيات كل تطبيق في Supabase أوضح وأأمن). لوحة الإدارة أصلًا منصة ويب منفصلة (Next.js) وليست تطبيق جوال. الثلاثة يتشاركون نفس قاعدة بيانات Supabase وبعض المكونات البرمجية (Design System)، لكنهم يُبنون ويُنزّلون كتطبيقات/منصات مستقلة تمامًا.

| النظام | النوع | المستخدم | الهدف الرئيسي | اللون التعريفي المقترح |
|---|---|---|---|---|
| تطبيق الراكبة | تطبيق جوال مستقل | المستخدمة النهائية (الركاب) | طلب رحلة، تتبعها، الدفع، استخدام المساعد الذكي | أرجواني دافئ (Purple) |
| تطبيق السائقة | تطبيق جوال مستقل | السائقات المعتمدات | استقبال الطلبات، رفع المستندات (KYC)، متابعة الأرباح | أزرق داكن (Deep Blue) |
| لوحة تحكم الإدارة | منصة ويب مستقلة | فريق التشغيل | تفعيل السائقات، مراقبة الرحلات، إدارة التسعير والتقارير | رمادي أنثراسايت مع لمسة ذهبية (Anthracite + Gold) |

> الفكرة: الهوية البصرية العامة لمشروع "أمانة" (الخط، الأيقونات، شكل الأزرار) تبقى موحّدة بين الثلاثة، لكن **اللون الأساسي (Primary Color)** يتغير حسب التطبيق — بهذا الشكل يعرف أي عضو بالفريق فورًا أي تطبيق يشتغل عليه، ويسهل التمييز بينها في العروض التقديمية ولقطات الشاشة.

**التقنيات المعتمدة:** React Native + Expo (تطبيقات الجوال) · Supabase (قاعدة البيانات + Auth + تخزين المستندات) · Next.js (لوحة الإدارة) · Vercel (استضافة الـ Backend والإدارة) · Expo EAS (توزيع تطبيقات الجوال).

**آلية التسجيل المعتمدة الآن:** بريد إلكتروني + كلمة مرور (عبر Supabase Auth)، مع شاشة "نسيت كلمة المرور" وتحقق من البريد (Email Verification) كخطوة تالية بعد التسجيل مباشرة، لكل من الراكبة والسائقة.

## 2. البرومبت الاحترافي لتوليد التصميم (Stitch)

استخدمي هذا البرومبت كنقطة بداية داخل Stitch، وعدّليه حسب كل تطبيق (استبدلي الجزء بين قوسين [ ] باسم التطبيق والشاشات المطلوبة):

```
أنت مصمم UI/UX محترف متخصص في تطبيقات الخدمات في السوق السعودي.
صمم واجهات لتطبيق [اسم التطبيق: الراكبة / السائقة / لوحة الإدارة] ضمن مشروع "أمانة"،
وهو منصة تنقّل ذكية مخصصة لتمكين المرأة في المملكة العربية السعودية،
متوافقة مع رؤية 2030.

المتطلبات العامة للتصميم:
- الاتجاه: من اليمين لليسار (RTL) بالكامل، مع دعم اللغة العربية كلغة أساسية.
- الهوية البصرية: هادئة، آمنة، أنيقة، تعكس الثقة والخصوصية، وتجنب الألوان الصاخبة.
  اللون الأساسي (Primary Color) لهذا التطبيق تحديدًا هو: [اكتبي هنا اللون من الجدول:
  أرجواني للراكبة / أزرق داكن للسائقة / رمادي أنثراسايت + ذهبي للإدارة]. استخدمي هذا
  اللون في الأزرار الرئيسية، العناصر النشطة (Active states)، والشريط العلوي، مع الحفاظ
  على خط وأيقونات موحّدة عبر التطبيقات الثلاثة لضمان انتماءها لعائلة تصميم واحدة.
- نظام التسجيل: بريد إلكتروني وكلمة مرور فقط في هذه المرحلة (بدون OTP/جوال).
- المكونات: أزرار كبيرة سهلة اللمس، حقول إدخال واضحة، رسائل خطأ ودودة،
  حالات تحميل (loading/empty states) لكل شاشة.
- التصميم متوافق مع أنظمة iOS وAndroid (Design System موحّد قابل لإعادة الاستخدام
  في React Native).

الشاشات المطلوبة لهذا التطبيق:
[أدرجي هنا قائمة الشاشات كما هي في القسم 3 أدناه، شاشة شاشة]

لكل شاشة، أعطني:
1. تخطيط بصري كامل (Layout) بدقة موبايل قياسية.
2. حالة افتراضية + حالة تحميل + حالة خطأ (إن وجدت).
3. تسلسل منطقي للانتقال بين الشاشات (User Flow).
```

## 4. البنية التقنية والبدء بالتنفيذ

### 4.1 مشروع واحد أم عدة مشاريع؟

**مشروع واحد (Monorepo) يحوي ثلاثة تطبيقات مستقلة داخله**، وليس ثلاثة مشاريع منفصلة على GitHub. هذا هو المعتمد في الشركات الاحترافية لهذا النوع من المنصات:

```
amana-platform/                 ← مستودع Git واحد على GitHub
├── apps/
│   ├── passenger/               ← تطبيق الراكبة (Expo/React Native)
│   ├── driver/                  ← تطبيق السائقة (Expo/React Native)
│   └── admin/                   ← لوحة الإدارة (Next.js)
├── packages/
│   ├── shared-ui/                ← مكونات تصميم مشتركة (أزرار، حقول، الألوان)
│   ├── shared-types/              ← أنواع TypeScript المشتركة (Ride, User, Driver...)
│   ├── supabase-client/            ← إعداد اتصال Supabase موحّد
│   └── i18n/                       ← ملفات الترجمة (ar.json / en.json)
├── package.json                   ← إدارة الحزم عبر npm/pnpm workspaces
└── turbo.json                     ← (اختياري) Turborepo لتسريع البناء
```

**كل تطبيق يُبنى ويُنشر بشكل مستقل تمامًا رغم وجوده في نفس المستودع:**
- تطبيق الراكبة وتطبيق السائقة → لا "يُستضافان" كالمواقع، بل يُبنيان عبر **EAS Build** وتُنشران على **App Store** و **Google Play** كتطبيقين منفصلين تمامًا، لكل منهما اسمه وأيقونته الخاصة.
- لوحة الإدارة (Next.js) → تُستضاف على **Vercel** فقط، كموقع ويب عادي تمامًا مثل مشاريعك السابقة.
- **قاعدة بيانات واحدة** على Supabase تخدم الثلاثة، مع صلاحيات (Row Level Security) تمنع أي تطبيق من الوصول لبيانات لا يفترض أن يراها (مثلًا: السائقة ما تشوف بيانات دفع الراكبة الكاملة).

### 4.2 دعم العربي/الإنجليزي والوضع الفاتح/الداكن

- **اللغة:** مكتبة `react-i18next` (أو `expo-localization` + i18next) مع ملفي ترجمة `ar.json` و`en.json` داخل `packages/i18n`. الاتجاه (RTL/LTR) يتغير تلقائيًا عبر `I18nManager` في React Native حسب اللغة المختارة.
- **الوضع الفاتح/الداكن:** استخدام `nativewind` (نسخة Tailwind لـ React Native) مع دعم `dark:` تلقائيًا حسب إعداد الجهاز، أو Context خاص يسمح للمستخدمة بالتبديل يدويًا وحفظ الاختيار محليًا.
- كلا الأمرين (اللغة والوضع) يُفضّل تصميمهما في Stitch من البداية — أضيفي لكل شاشة رئيسية نسخة عربية/فاتح كحد أدنى، والباقي يُبنى برمجيًا بنفس المكونات.

### 4.3 الحسابات المطلوب إنشاؤها

| الحساب | الغرض | يُستخدم من |
|---|---|---|
| **GitHub** | استضافة الكود (Monorepo) | الفريق كامل |
| **Supabase** | قاعدة البيانات + المصادقة (Auth) + تخزين الملفات — **وليس Firebase**، المشروع معتمد بالكامل على Supabase | الثلاثة تطبيقات |
| **Vercel** | استضافة لوحة الإدارة (Next.js) فقط | لوحة الإدارة فقط |
| **Expo (EAS)** | بناء تطبيقي الجوال ونشرهما على المتاجر | الراكبة والسائقة |
| **Apple Developer** (لاحقًا) | نشر التطبيقين على App Store | عند مرحلة الإطلاق فقط |
| **Google Play Console** (لاحقًا) | نشر التطبيقين على Google Play | عند مرحلة الإطلاق فقط |

> لا حاجة لأي حساب Firebase — التسجيل والمصادقة وقاعدة البيانات كلها عبر Supabase.

### 4.4 كيف يتصل التطبيق بقاعدة البيانات؟

كل تطبيق (الراكبة، السائقة، الإدارة) يتصل بـ Supabase **مباشرة** عبر مكتبة `@supabase/supabase-js`، بنفس الطريقة تمامًا التي اعتدتها في مشاريع الويب — الفرق الوحيد أن الاتصال الآن يحصل داخل تطبيق React Native بدل موقع ويب:

- كل تطبيق يحمل `SUPABASE_URL` و `SUPABASE_ANON_KEY` (مفتاح عام آمن للاستخدام في التطبيقات).
- المصادقة (تسجيل الدخول بالبريد وكلمة المرور) تتم عبر `supabase.auth.signInWithPassword()`.
- العمليات الحساسة فقط (مثل تفعيل حساب سائقة من الإدارة) تمر عبر **API Routes/Server Actions في Next.js** تستخدم `SERVICE_ROLE_KEY` — وهذا المفتاح **لا يوضع أبدًا داخل تطبيقات الجوال**، يبقى على السيرفر فقط (Vercel).
- لا حاجة لبناء سيرفر Backend منفصل — Supabase يقوم بدور الـ Backend الكامل (قاعدة بيانات + Auth + تخزين + Realtime لتتبع الرحلات لحظيًا).

### 4.5 اسم المجلد المحلي والإصدار

- **اسم المجلد المقترح على جهازك:** `amana-platform`
- **إصدار Expo:** استخدمي أحدث إصدار مستقر حاليًا وهو **Expo SDK 57** (مبني على React Native 0.86)، عبر أمر الإنشاء القياسي `npx create-expo-app`.

### 4.6 برومبت للبدء بالتنفيذ عبر Claude Code

```
أريد إنشاء مشروع Monorepo باسم "amana-platform" لمنصة "أمانة" - منصة تنقّل ذكية
للمرأة في السعودية. أعطني الهيكل الكامل والإعداد الأولي بالمواصفات التالية:

البنية:
- Monorepo بإدارة npm workspaces (أو pnpm workspaces إن كان أفضل)
- apps/passenger و apps/driver: مشروعا Expo (أحدث SDK مستقر) بـ TypeScript
  و Expo Router
- apps/admin: مشروع Next.js (أحدث إصدار مستقر) بـ TypeScript و App Router
- packages/shared-ui, packages/shared-types, packages/supabase-client,
  packages/i18n كحزم مشتركة بين التطبيقات

المتطلبات التقنية:
- التسجيل والدخول عبر Supabase Auth بالبريد الإلكتروني وكلمة المرور فقط
  (signUp / signInWithPassword)، مع شاشات: تسجيل دخول، إنشاء حساب،
  نسيت كلمة المرور، تحقق من البريد
- إعداد عميل Supabase موحّد في packages/supabase-client يُستخدم من التطبيقات
  الثلاثة، مع فصل واضح بين المفتاح العام (anon key) الذي يُستخدم في التطبيقات
  ومفتاح service role الذي يبقى فقط في Server Actions بمشروع admin
- دعم اللغتين العربية (افتراضية) والإنجليزية عبر react-i18next، مع تبديل
  تلقائي لاتجاه الواجهة RTL/LTR
- دعم الوضع الفاتح والداكن في التطبيقات الثلاثة باستخدام nativewind
- نظام ألوان أساسي مختلف لكل تطبيق: أرجواني للراكبة، أزرق داكن للسائقة،
  رمادي أنثراسايت + ذهبي للإدارة — كمتغيرات CSS/Theme قابلة لإعادة الاستخدام
- ملف .env.example في كل تطبيق يوضح المتغيرات البيئية المطلوبة (بدون قيم حقيقية)
- ملف README.md في جذر المشروع يشرح طريقة تشغيل كل تطبيق محليًا

ابدأ بإنشاء هيكل المجلدات والملفات الأساسية فقط (بدون منطق أعمال معقد بعد)،
مع تهيئة TypeScript و ESLint و Prettier في كل تطبيق، ثم أخبرني بالخطوات
التالية المقترحة بعد المراجعة.
```

## 5. الحالة الحالية للمشروع (سياق لكلود كود)

هذا القسم يلخّص ما تم إنجازه فعليًا حتى الآن، حتى يبدأ كلود كود من حيث توقف العمل دون إعادة شرح:

- أكمل ربط حساب nuwate369 عبر SSH alias المُعد مسبقًا.

  Supabase الذي سأنشئه، اربطها في ملفات .env المناسبة لكل تطبيق
